# 1.3. Módulo Modelagem BPMN

## Introdução
A modelagem BPMN (*Business Process Model and Notation*) é uma notação padrão para representar os processos de trabalho de forma visual e compreensível, facilitando o alinhamento entre a equipe e os stakeholders. Para o desenvolvimento do **MinhaTirinha**, o Grupo 08 adotou uma abordagem baseada em metodologias ágeis, focando na rastreabilidade e qualidade técnica.

## Metodologia
O grupo optou por uma metodologia híbrida, integrando o framework **Scrum** para a gestão e planejamento das atividades com práticas do **Extreme Programming (XP)** para garantir a qualidade da engenharia de software:

* **Scrum (Gestão):** Utilizamos ciclos de desenvolvimento de duas semanas (**Sprints**), iniciados com o **Sprint Planning** para definir o que será feito e encerrados com a **Sprint Review** e **Retrospective** para avaliar o incremento e o processo.
* **XP (Engenharia):** Aplicamos práticas de **Programação em Par** (*Pair Programming*) para a codificação e revisão constante de código através de **Pull Requests** no GitHub, além de focar em **Integração Contínua (CI)**.
* **Gestão Visual:** Utilizamos o **Trello** para o controle do fluxo de tarefas, permitindo que todos os membros acompanhem o estado das atividades (Backlog, Doing, Done).

### Explicação do Fluxo Gerado

O processo foi modelado em duas raias principais para separar as responsabilidades de gestão e execução técnica:

1.  **Ciclo de Gestão (Scrum):**
    * O fluxo inicia no **Product Backlog**, onde as necessidades do projeto são mapeadas.
    * No **Sprint Planning**, as tarefas são priorizadas e movidas para o **Sprint Backlog (Trello)**.
    * O ciclo é retroalimentado pela **Daily Scrum** (sincronização diária) e finaliza com a **Sprint Review** e **Retrospective**, onde o incremento funcional é validado antes de ser considerado "Done".

2.  **Ciclo Técnico de Desenvolvimento (XP & Trello):**
    * **Puxar Tarefa:** Os desenvolvedores retiram as tarefas do Trello e iniciam a **Programação em Par** (Pair Programming).
    * **TDD (Test Driven Development):** Aplicamos o ciclo *Red-Green-Refactor* (Escrever teste -> Falhar -> Escrever código -> Passar -> Refatorar).
    * **CI/CD e Code Review:** Após a criação do **Pull Request** no GitHub, são executados testes de integração. Um colega realiza o **Code Review** síncrono para garantir que o código segue os padrões do grupo.
    * **Entrega:** Somente após a aprovação técnica a tarefa é movida para a coluna **"Done"** no Trello, gerando o incremento funcional da Sprint.
